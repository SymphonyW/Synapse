# 总体架构

## 1. 系统目标

Synapse 的目标是提供一条可观测、可取消、可重试、可扩展的 Agent 任务执行链路，覆盖从任务提交到结果回传的完整生命周期。

## 2. 组件边界

1. Web 控制台：位于 [apps/web](../apps/web)，负责用户交互、会话管理、运维操作。
2. Gateway 服务：位于 [services/gateway-go](../services/gateway-go)，负责 API、权限、任务编排、事件流输出。
3. AI Engine：位于 [services/ai-engine-py](../services/ai-engine-py)，负责模型调用与 token 流生成。
4. 协议层：位于 [proto/synapse/v1/agent.proto](../proto/synapse/v1/agent.proto)，定义 gRPC 交互契约。
5. 数据与消息基础设施：PostgreSQL + Redis（不可用时可回退内存实现）。

## 3. 关键数据流

1. 用户在 Web 端提交任务（HTTP）。
2. Gateway 校验会话与权限，创建任务并入队。
3. Worker 出队调用 AI Engine.SubmitTask（gRPC 流）。
4. AI Engine 输出 started/info/token/completed/failed 事件；当遇到审批门禁时会输出 approval_required 信息。
5. Gateway Worker 解析 approval_required 后把任务切换为 paused，并持久化 paused 事件。
6. Gateway 持久化任务事件并通过 SSE 对前端增量输出。
7. 前端基于 last_event_id 续传机制持续更新界面。

## 4. 设计特点

1. 双层流式：内部 gRPC 流 + 外部 SSE 流。
2. 存储与队列可插拔：Postgres/Redis 与内存实现可切换。
3. 取消语义清晰：queued/running/paused 可取消，终态冲突保护。
4. 对话上下文后端构建：避免前端拼接 prompt 不一致。
5. 审批恢复闭环：paused -> approve -> queued -> 继续执行。

## 5. 可用性策略

1. Postgres 初始化失败时，自动回退 [services/gateway-go/internal/store/inmemory.go](../services/gateway-go/internal/store/inmemory.go)。
2. Redis 初始化失败时，自动回退 [services/gateway-go/internal/queue/inmemory.go](../services/gateway-go/internal/queue/inmemory.go)。
3. AI Runtime 支持重试与降级，降低上游波动影响。

## 6. 主要风险点

1. 生产环境若使用内存回退会导致重启丢数据。
2. 单实例 Worker 可能成为吞吐瓶颈。
3. 会话上下文增长需要持续治理（窗口与摘要策略）。
