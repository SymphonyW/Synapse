# 协议与通信

Synapse 使用两层协议：内部服务间使用 gRPC streaming，浏览器对外使用 HTTP JSON 和 SSE。

## gRPC 契约

主协议文件：[proto/synapse/v1/agent.proto](../proto/synapse/v1/agent.proto)。

| RPC | 方向 | 说明 |
|---|---|---|
| `Health(HealthRequest) returns HealthResponse` | Gateway -> AI Engine | 健康检查，返回 status 和 model_provider |
| `SubmitTask(SubmitTaskRequest) returns stream AgentEvent` | Gateway -> AI Engine | 提交任务并流式接收事件 |
| `MemoryWrite(MemoryWriteRequest) returns MemoryWriteResponse` | Gateway -> AI Engine | 写入长期记忆 |
| `MemoryRecall(MemoryRecallRequest) returns MemoryRecallResponse` | Gateway -> AI Engine | 召回长期记忆 |
| `MemoryDelete(MemoryDeleteRequest) returns MemoryDeleteResponse` | Gateway -> AI Engine | 删除长期记忆 |
| `MemoryList(MemoryListRequest) returns MemoryListResponse` | Gateway -> AI Engine | 列出长期记忆 |

## SubmitTaskRequest

| 字段 | 说明 |
|---|---|
| `task_id` | Gateway 生成的任务 ID |
| `user_id` | 会话用户名，普通用户不能伪造 |
| `prompt` | 当前轮用户输入 |
| `metadata` | 控制 Agent、会话、审批、记忆和权限的扩展字段 |

常见 metadata：

| Key | 说明 |
|---|---|
| `conversation_id` | 会话 ID |
| `user_message` | 当前用户消息 |
| `client_view` | 前端视图，`chat` 会触发会话上下文构建 |
| `model_messages_json` | Gateway 构造的 OpenAI messages JSON |
| `model_prompt` | Gateway 构造的文本 prompt 兼容字段 |
| `auth_user_role` | Gateway 注入的角色 |
| `auth_username` | Gateway 注入的用户名 |
| `agent_enabled` | 是否启用 Agent loop |
| `memory_write_enabled` | 是否自动写入长期记忆 |
| `approval_granted` | 是否为审批恢复任务 |
| `approved_tools` | 旧版工具名级审批，逗号分隔 |
| `approved_tool_call` | 精确审批记录，包含工具名、输入、风险和恢复步点 |
| `agent_resume_step_index` | Agent 恢复执行步点 |

## AgentEvent

| proto 枚举 | Gateway/SSE 类型 | 说明 |
|---|---|---|
| `AGENT_EVENT_TYPE_STARTED` | `started` | AI Engine 开始执行任务 |
| `AGENT_EVENT_TYPE_TOKEN` | `token` | 增量文本 |
| `AGENT_EVENT_TYPE_INFO` | `info` | 结构化 Agent 信息或普通诊断信息 |
| `AGENT_EVENT_TYPE_COMPLETED` | `completed` | AI Engine 正常完成 |
| `AGENT_EVENT_TYPE_FAILED` | `failed` | AI Engine 执行失败 |

Gateway 派生事件：

| 事件 | 来源 |
|---|---|
| `paused` | Worker 解析 `approval_required` 后写入 |
| `approval_granted` | `/approve` 成功后写入 |
| `resume_requested` | `/approve` 成功后写入 |
| `cancel_requested` | `/cancel` 成功后写入 |
| `canceled` | Worker 或 API 终止任务后写入 |
| `dead_lettered` | 重试耗尽后写入 |
| `replay_requested` | `/replay` 成功后写入 |
| `terminal` | SSE 层在终态无新事件后发送，不入库 |

## Agent info 结构

AI Engine 的 `info` message 通常是 JSON 字符串：

```json
{
  "schema": "synapse.agent.info.v1",
  "agent_event": "tool_finished",
  "payload": {
    "tool": "calculator",
    "ok": true
  },
  "display_message": "Tool finished: calculator"
}
```

常见 `agent_event`：

| 事件 | 说明 |
|---|---|
| `perceive` | 读取任务和上下文 |
| `memory_recall` | 长期记忆召回 |
| `plan` | 规划步骤 |
| `act` | 执行步骤 |
| `tool_selected` | 选择工具 |
| `tool_started` | 开始工具调用 |
| `tool_finished` | 工具调用成功 |
| `tool_failed` | 工具调用失败 |
| `tool_skipped` | 无工具或策略阻断 |
| `approval_required` | 需要人工审批 |
| `replan` | 失败后重规划 |
| `observe` | 观察执行结果 |
| `reflect` | 反思当前步骤 |
| `memory_write` | 写入长期记忆 |
| `evaluate` | 评估任务完成度 |

## SSE 协议

接口：

```text
GET /v1/tasks/{taskID}/events?last_event_id=<id>
```

行为：

1. 响应 `Content-Type: text/event-stream`。
2. 首帧发送 `info`，表示 stream 已打开。
3. 后续从 TaskStore 轮询 `id > last_event_id` 的事件。
4. 每个事件 payload 包含 `event_id`、`type`、`message`、`token`、`trace_id`、`emitted_at_unix_ms`。
5. 任务进入 completed/failed/canceled 后，若无新事件，发送 `terminal` 并关闭。
