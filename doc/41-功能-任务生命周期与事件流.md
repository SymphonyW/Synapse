# 功能：任务生命周期与事件流

## 1. 功能目标

为任务提供可追踪、可恢复的状态机，并通过事件流驱动前端实时反馈。

## 2. 状态定义

状态定义位于 [services/gateway-go/internal/domain/task.go](../services/gateway-go/internal/domain/task.go)：

1. queued
2. running
3. paused
4. completed
5. failed
6. canceled

## 3. 典型状态迁移

1. create：queued
2. worker 开始执行：running
3. 触发审批门禁：paused
4. 人工审批恢复：queued（重新入队）
5. 正常完成：completed
6. 异常失败：failed
7. 用户取消：canceled

## 4. 事件类型

常见事件：

1. info
2. started
3. token
4. paused
5. approval_granted
6. resume_requested
7. cancel_requested
8. canceled
9. completed
10. failed
11. dead_lettered
12. replay_requested
13. terminal

## 5. SSE 机制

接口：GET /v1/tasks/{taskID}/events

机制：

1. 支持 last_event_id 增量拉取。
2. 每条事件包含 event_id。
3. 终态且无新事件时发送 terminal。

## 6. 取消语义

1. queued/running/paused 首次取消：202。
2. 已取消再次取消：200（幂等）。
3. completed/failed 不可取消：409。

## 7. 审批恢复语义

1. Worker 解析 info 事件中的 approval_required 后，任务切换为 paused。
2. API 调用 POST /v1/tasks/{taskID}/approve 后，任务切换为 queued 并重新入队。
3. 网关会追加 approval_granted 与 resume_requested 事件。
4. 仅 paused 任务允许审批恢复，非 paused 返回 409。

## 8. 前端消费要点

1. 维护 task 级 last_event_id 游标。
2. 去重 event_id，防止重连重复渲染。
3. token 事件直接拼接到输出缓冲。
4. 对 paused/approval_granted/resume_requested 事件给出显式运维动作提示。

## 9. 监控建议

1. 统计各状态数量和转化率。
2. 统计任务平均耗时与 P95。
3. 统计 token 事件吞吐与 SSE 活跃连接数。
4. 统计 paused 数量、审批恢复耗时和审批后成功率。


## 10. Replay 关系

1. `POST /v1/tasks/{taskID}/replay` 现在创建新的子任务，而不是覆盖原任务。
2. 子任务通过正式字段 `replay_of_task_id` 指向原任务；原任务通过 `GET /v1/tasks/{taskID}/replays` 查询全部 replay 子任务。
3. `replay_requested` 事件仍写回原任务，便于审计“何时发起了新的 replay”。
4. 这种做法让原 trace 与 replay trace 可以并存，支撑后续结构化 diff。
