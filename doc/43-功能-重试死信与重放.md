# 功能：重试、死信与重放

## 1. 功能目标

提高任务执行链路在外部不稳定场景下的韧性，避免单次失败直接导致任务不可恢复。

## 3. 重试机制

1. 最大重试次数：SYNAPSE_TASK_MAX_ATTEMPTS。
2. 重试间隔：SYNAPSE_TASK_RETRY_BACKOFF。
3. 重试过程中写入 info 事件，便于可观测。
4. 不可重试错误快速失败，减少无效消耗。

## 4. 不可重试错误示例

1. DeadlineExceeded
2. InvalidArgument
3. PermissionDenied
4. Unauthenticated
5. Unimplemented
6. FailedPrecondition

## 5. 死信机制

1. 最后一轮重试失败后写入 dead_letter_tasks。
2. 追加 failed 与 dead_lettered 事件。
3. 运维台可查询死信列表并发起重放。

## 6. 重放机制

1. ReplayTask 仅禁止 running 任务重放；paused 任务也可重放。
2. 状态重置为 queued。
3. 清理死信记录。
4. 追加 replay_requested 事件。
5. 重新入队等待执行。

## 7. 取消与死信关系

1. 任务进入取消流程时会清理死信记录。
2. finalizeCanceled 优先保留已有取消原因。
3. 取消不属于失败重试路径。

## 8. 审批暂停与死信关系

1. approval_required 会把任务切换到 paused，不会直接进入 failed/dead_letter。
2. paused 任务可走 approve 恢复执行，也可走 replay 重新启动。
3. 只有重试耗尽且最终失败时才会进入 dead_lettered。

## 9. 运维建议

1. 对 dead_lettered 事件设置告警。
2. 定期分析死信原因分布（provider 失败、配置错误、权限错误）。
3. 对高频失败任务引入自动熔断或降级策略。


## 10. Replay 子任务与比较

1. replay 仍禁止 `running` 任务触发，但现在会创建新的 `queued` 子任务，原任务保持原状态与原事件不变。
2. replay 子任务通过正式字段 `replay_of_task_id` 指向源任务。
3. `GET /v1/tasks/{taskID}/replays` 返回源任务的 replay 子任务列表。
4. `GET /v1/tasks/{taskID}/compare/{otherTaskID}` 仅允许比较源任务及其直接 replay 子任务，并返回两侧任务快照与事件快照。
5. 这种关系建模避免了“原任务被重放覆盖后无法再比较”的问题。
