# 功能：审批暂停与恢复

## 1. 功能目标

当 Agent 执行触发高风险工具且缺少审批时，系统不直接失败，而是进入可恢复的 paused 状态，允许人工审批后继续执行。

## 3. 触发条件

1. AI Engine 在运行循环中检测到工具审批条件不满足。
2. Runtime 输出 info 事件，agent_event=approval_required，payload 中包含 tool、tool_input、risk_level、resume_step_index 与 approved_tool_call。
3. Gateway Worker 解析该 info 后执行暂停迁移。

## 4. 状态与事件语义

1. 状态迁移：running -> paused。
2. Worker 会写入 paused 事件，message 形如 task paused: approval required for tool <tool>。
3. 审批恢复后状态迁移：paused -> queued（重新入队）。
4. 审批恢复时会追加 approval_granted 与 resume_requested 事件。

## 5. 关键元数据约定

1. approval_granted：是否授予审批（true/false）。
2. approved_tools：允许恢复时可用工具列表（逗号分隔）。
3. approved_tool_call：精确审批记录，包含工具名、输入、风险等级、原因和恢复步点。
4. agent_resume_step_index：恢复执行步点。
5. agent_required_tool：触发暂停所需的工具名。
6. agent_required_tool_input：触发暂停的工具输入。
7. agent_required_tool_risk_level：触发暂停的工具风险等级。
8. agent_required_reason：触发暂停原因。
9. agent_resume_requested_by：执行恢复操作的用户。

## 6. 审批接口

接口：POST /v1/tasks/{taskID}/approve

请求体（可选字段）：

1. requested_by
2. reason
3. approved_tools
4. approved_tool_call

推荐请求体：

```json
{
  "reason": "manual approval",
  "approved_tool_call": {
    "tool_name": "summarize_page",
    "tool_input": "https://example.com",
    "risk_level": "high",
    "reason": "manual approval",
    "resume_step_index": 1
  }
}
```

返回码：

1. 202：审批已接受，任务已重置为 queued 并入队。
2. 403：无权限（非 owner/admin）。
3. 404：任务不存在。
4. 409：任务不是 paused，不能审批恢复。
5. 500：元数据写入或入队失败。

## 7. 前端运维流程

1. 运维台筛选到 paused 任务。
2. 在任务详情区点击 Approve & Resume。
3. 前端调用 approve 接口并优先附带 approved_tool_call；旧任务可退回 approved_tools。
4. 恢复后任务进入 queued/running，事件流继续输出。

## 8. 设计价值

1. 把“高风险阻断”从硬失败改为可审计、可恢复流程。
2. 任务上下文和历史事件可完整保留，减少重复执行损耗。
3. 审批动作可追踪到执行人和时间点，便于运维治理。

## 9. 风险与建议

1. 不建议长期使用全局 approval_granted=true，建议优先使用 approved_tool_call 做参数级最小授权。
2. 建议对 paused 停留时长建立告警，避免任务长期堆积。
3. 建议把审批事件纳入审计报表（审批人、恢复成功率、恢复耗时）。
