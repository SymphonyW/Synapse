# Agent Trace 工作台

Agent Trace 工作台用于把一次任务从“感知到输出”的链路从事件流水提升为可读的执行叙事。它不改变后端协议，也不改写 AI Engine 事件格式，而是在 Web 前端把已有标准化 `info` 事件重新建模成结构化视图。

| 区域 | 展示内容 |
|---|---|
| 任务信息 | `task id`、`conversation id`、任务状态 |
| 前置阶段 | `perceive`、`memory_recall`、`plan` |
| 执行阶段 | 每个 step、工具选择、工具开始、工具完成/失败、审批节点、重规划 |
| 后置阶段 | `observe`、`reflect`、`synthesis_mode`、`memory_write`、`evaluate` |
| 诊断摘要 | 工具调用数、成功/失败数、是否审批暂停、是否重规划、最后失败原因 |

## 适合的场景

| 场景 | 为什么有用 |
|---|---|
| 调试 | 快速看出失败发生在哪一步、哪个工具、是否发生 fallback |
| 演示 | 用阶段化时间线解释 Agent 如何工作，而不是只展示最终答案 |
| 排障 | 结合审批、重规划和 evaluate 指标，定位“停住”“降级”“失败”的原因 |
| 审计 | 保留原始事件 JSON，便于开发者继续向下追查 |

## 两种视图

| 视图 | 用途 |
|---|---|
| 结构化 Trace | 日常查看，按阶段、step 和工具调用理解执行链路 |
| 原始事件 JSON | 开发排查，保留事件原貌并支持复制 |

如果某条事件 JSON 解析失败，工作台会继续保留原始事件视图，并尽量展示其余已成功解析的阶段，避免页面因为单条坏事件而白屏。

## 导出内容

当前任务可导出为 `synapse-trace-task-{taskId}.json`，内容包括：

1. `task_id`
2. `conversation_id`
3. `status`
4. `raw_events`
5. `parsed_trace_summary`
6. `exported_at`

这份导出文件适合附在问题单、演示材料或复盘记录中。

## 如何理解 approval 与 replan

| 事件 | 含义 |
|---|---|
| `approval_required` | 当前工具调用因风险策略需要人工确认；任务可能进入 paused，继续执行前需审批恢复 |
| `replan` | 某个 step 的原始执行路径失败后，Agent 选择新的工具或路径继续尝试 |

在工作台里，这两个节点都会高亮，因为它们通常正是“为什么任务停下”或“为什么链路发生转向”的关键。

## Replay Diff

Trace 工作台现在可以直接查看某任务的 replay 记录，并复用现有 `traceParser.ts` 生成结构化 diff：

1. 顶部差异摘要：状态、总耗时、计划步骤数、工具调用数量、审批、重规划、记忆召回命中和最终回答长度。
2. 左右并排比较：左侧原任务，右侧 replay 子任务。
3. 折叠分段：阶段完整性、工具调用序列、`evaluate` 指标、最终回答文本 diff。
4. 新增/缺失工具调用会高亮；若 trace 不完整，页面仍显示已知差异并明确提示。
