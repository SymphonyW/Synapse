# 后续开发任务清单

本文将后续开发拆成可验收任务。编号中的 P0/P1/P2 表示建议优先级，不代表必须一次性完成全部内容。

| 编号 | 任务名 | 目标 | 主要改动模块 | 依赖前置 | 建议验收方式 |
|---|---|---|---|---|---|
| P0-01 | Web 工程化拆分 | 将 `App.tsx` 中 API、SSE、认证、聊天、运维、Agent 时间线拆成可维护模块，为后续页面扩展打底 | `apps/web/src/App.tsx`、新增 `apps/web/src/api`、`hooks`、`components` | 不改后端接口；先锁定现有 UI 行为 | `npm run build` 通过；核心操作仍可完成登录、创建任务、SSE、审批、取消、死信重放 |
| P0-02 | 记忆管理 UI | 在 Web 增加长期记忆列表、召回、手工写入、删除入口，复用已有 Gateway Memory API | `apps/web/src`、`services/gateway-go/internal/api/handlers_memory.go` 只读确认 | P0-01；后端 `/v1/memories` 已可用 | 使用普通用户验证只看到自己的记忆；管理员带 `user_id` 验证跨用户查询；删除后列表不再出现 |
| P0-03 | Agent Trace 工作台 MVP | 基于 `task_events` 展示单任务完整 Agent 阶段、工具调用、审批、记忆和最终回答 | `apps/web/src`、`services/gateway-go/internal/api/handlers.go` 可选增加只读查询参数 | P0-01；现有事件语义稳定 | 用包含 `memory_recall`、`tool_finished`、`approval_required` 的任务验证时间线；刷新后仍能从事件恢复 |
| P0-04 | README 与 Demo 包装 | 整理最短启动、演示脚本、截图/录屏路径和常见场景，让项目可被快速验收 | `README.md`、`doc/05-接口验证手册.md`、可选 `scripts/dev.ps1` 文档说明 | 功能行为不变；先不新增运行时接口 | 新开发者按 README 可在 mock 模式跑通登录、创建任务、SSE、审批、记忆 API |
| P1-01 | 工具策略管理中心 | 将当前环境变量策略可视化，先提供策略查看、校验和说明，再规划持久化策略 | `apps/web/src`、`services/ai-engine-py/app/tools/policy.py`、未来 Gateway 策略 API | P0-03；明确策略是否仍由 AI Engine 生效 | UI 能展示角色允许工具、审批工具、禁用工具；错误 JSON 能给出明确校验失败原因 |
| P1-02 | OpenAPI 真实执行器 | 为 `OpenAPIToolProvider` 补受控 HTTP executor，使 OpenAPI 工具能真实调用 allowlist API | `services/ai-engine-py/app/tools/providers.py`、`runtime.py`、测试 | 工具策略和审计链路不能绕过；先定义 allowlist、auth、超时 | 新增单元测试覆盖 GET/POST JSON、缺失 auth、非 allowlist；回归评测新增 OpenAPI executor 用例 |
| P1-03 | MCP 真接入 | 用现有 `MCPAdapter`/`MCPToolProvider` 接入真实 MCP transport，并保留统一治理 | `services/ai-engine-py/app/tools/providers.py`、新增 MCP adapter 模块、配置 | 明确 stdio 或 HTTP transport MVP；工具策略中心至少能解释 MCP 工具来源 | fake adapter 测试保留；新增真实 stdio MCP smoke test；审批和审计日志能记录 `mcp_*` 工具 |
| P1-04 | Replay Diff | 对重放前后事件、最终回答、工具调用和失败原因做结构化对比 | Gateway 只读事件查询、`apps/web/src` Trace 页面、可能新增 AI Engine benchmark 辅助 | P0-03；重放事件归档策略明确 | 对同一任务执行 replay 后能看到 token diff、工具序列差异、状态耗时差异 |
| P1-05 | 真实模型 Benchmark | 在 mock regression 外增加真实 provider 分层评测，记录质量、延迟、失败率和成本字段 | `services/ai-engine-py/app/benchmarks`、配置文档、脚本 | 已有 mock regression 必须保留；需要 provider key 管理策略 | 无 key 时跳过真实评测且返回明确状态；有 key 时输出 JSON 指标并不影响 mock 门禁 |
| P2-01 | 向量记忆 | 实现 `VectorMemoryStore`，在保持 Memory RPC 契约不变的前提下引入 embedding 和向量召回 | `services/ai-engine-py/app/memory.py`、配置、测试、可能新增存储服务 | 记忆 UI 已能展示召回解释；确定向量库或数据库方案 | 单元测试覆盖写入、召回、删除、回退；同一 API 能切换 file/vector backend |
| P2-02 | Web 数据层升级 | 引入统一数据请求层，降低轮询、SSE、状态同步的耦合 | `apps/web/src`，可选 React Query 或自研轻量 store | P0-01 后再做，避免和拆分冲突 | 任务列表、详情、死信、健康状态刷新行为保持一致；重复请求和错误处理集中 |
| P2-03 | 队列可靠性升级 | 替换或增强 Redis List，支持 ack/reclaim，避免 Worker 异常后任务丢失执行权 | `services/gateway-go/internal/queue`、`worker/processor.go`、文档 | 需要确定是否使用 Redis Streams 或外部 MQ | 模拟 Worker 崩溃后任务可被重新领取；取消、重试、死信语义不退化 |
| P2-04 | 数据库 migration | 将 `ensureSchema` 自动建表迁移为版本化 migration | `services/gateway-go/internal/store/postgres.go`、新增 migration 目录、启动脚本 | 明确本地和容器启动方式；保留开发友好路径 | 空库可初始化；旧库可升级；重复执行不破坏数据 |
| P2-05 | 接口文档生成 | 补 OpenAPI/Swagger 或 Apifox/Postman 集合，覆盖 Gateway HTTP API | `services/gateway-go/internal/api`、`doc`、脚本 | API 错误码和请求响应结构稳定 | 文档能覆盖认证、任务、SSE、审批、记忆、死信；示例请求可执行 |

## 分组建议

| 阶段 | 推荐范围 | 说明 |
|---|---|---|
| MVP | P0-01 到 P0-04 | 主要复用现有接口，目标是可维护、可演示、可调试 |
| 第一阶段 | P1-01 到 P1-05 | 围绕工具生态、Trace 和评测闭环扩展，需要更严格验收 |
| 第二阶段 | P2-01 到 P2-05 | 涉及存储、队列、生产化和真实模型成本，适合独立设计后推进 |
